# 프로젝트 회고 — CryptoVol Dashboard

> P학기 팀 프로젝트(GARCH 변동성 분석)를 풀스택 웹 서비스로 확장한 개인 프로젝트 회고

---

## 왜 만들었나

P학기에서 GARCH 모형으로 비트코인 변동성을 분석했지만, 결과물은 Jupyter Notebook에 머물렀다. "통계 모형을 실시간 서비스로 만들 수 있을까?"라는 질문에 답하기 위해, 데이터 수집 → 모형 예측 → 시각화까지 이어지는 파이프라인 전체를 직접 구현했다.

---

## 기술적 도전

| 도전 | 결정 | 돌아보면 |
|------|------|----------|
| **GARCH 모형 API 서빙** — 5개 모형 적합이 수백ms 소요 | 인메모리 캐싱 + 120일 윈도우 제한, 실패 시 0.0 반환으로 그레이스풀 디그레이션 | 캐시 무효화 전략(TTL)이 없어 오래된 예측이 남을 수 있음 |
| **WebSocket 릴레이** — 프론트에서 직접 Binance 연결 시 CORS/키 노출 | FastAPI가 Binance 스트림 수신 → Set 기반 클라이언트 브로드캐스트 | 고정 3초 재연결 대신 지수 백오프가 필요, 백프레셔 미처리 |
| **프론트엔드 상태 관리** — 26개 컴포넌트에 실시간 데이터 전달 | 외부 라이브러리 없이 Dashboard.jsx에서 useState 중앙 관리 | 규모가 커지면 Context API나 Zustand 도입 필요 |
| **데이터 백필** — 빈 DB로 시작하면 GARCH 예측 불가 | 서버 시작 시 365일 데이터 자동 백필 | 서버 시작이 블로킹됨 → 백그라운드 태스크 + health check가 더 나은 설계 |
| **Monte Carlo 시뮬레이션** — GARCH를 포트폴리오 리스크에 연결 | 1,000 시나리오 MC + VaR(95%/99%) + Sharpe Ratio 산출 | 1,000개는 99% VaR에 불충분, 정규분포 가정이 팻테일 미반영 |

## 배운 점

**1. 분석 코드와 서비스 코드는 다르다**
Jupyter에서 잘 돌아가는 GARCH가 API에서는 적합 시간, 예외 전파, 동시 요청 문제를 일으킨다. 각 모형을 독립적으로 try-except 처리하는 패턴을 익혔다.

**2. WebSocket은 HTTP와 다른 사고방식이 필요하다**
연결 수명주기, 재연결, 클라이언트 추적 — REST에 없던 상태 관리가 핵심이다. `useRef`로 연결 객체와 리렌더링을 분리하고, cleanup에서 타이머와 연결을 모두 정리해야 메모리 누수를 방지한다는 것을 배웠다.

**3. 모형의 한계를 인정하는 것도 설계다**
시장 급변 시 5개 모형이 동시에 틀린다. 예측 정확도 트래커를 만들어 모형 성능을 투명하게 보여준 것은 좋은 결정이었다. HAR-TGARCH-X에 외생변수가 실제로 전달되지 않는 구현 누락도 뒤늦게 발견했다 — 수학적 이해와 올바른 구현은 별개다.

**4. "지금 충분한 것"과 "나중에 필요한 것"의 경계**
순수 useState + 직접 작성한 번역 객체 + Tailwind만으로 26개 컴포넌트를 만들 수 있었지만, 규모가 커지면 기술 부채가 된다. 라이브러리 도입 시점을 판단하는 감각을 얻었다.

**5. 데이터 파이프라인 안정성 = 서비스 신뢰도**
CoinGecko Rate Limit 한 번이 DB 갭 → GARCH 왜곡으로 이어진다. 재시도 로직과 데이터 갭 감지가 프로덕션에서는 필수라는 것을 직접 경험했다.

## 이 프로젝트가 남긴 것

"통계 분석을 할 수 있다"에서 **"통계 분석을 서비스로 만들 수 있다"**로의 전환이었다. 분석가는 모형의 AIC를 비교하지만, 엔지니어는 그 모형이 API 타임아웃 안에 응답할 수 있는지를 고민한다. 이 차이를 체감한 프로젝트였고, 동시에 테스트, 캐싱, 모니터링 등 **프로덕션 품질**에 도달하려면 더 깊이 파야 할 영역이 명확해졌다.
